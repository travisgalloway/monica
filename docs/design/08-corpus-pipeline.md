# Corpus pipeline (datatrove + RunPod + R2)

[← Index](README.md)

How the scale-up corpus is built, stored, and post-trained. The offline
[`src/data/`](../../src/data/) stages ([data pipeline](04-data-pipeline.md)) are the
laptop-scale version of this; at scale the same shape runs on **Hugging Face
[`datatrove`](https://github.com/huggingface/datatrove)**, stores intermediates on
**Cloudflare R2**, and processes on **RunPod**. This doc is the decision record for that
flow; the GitHub tracker is [issue #65](https://github.com/travisgalloway/monica/issues/65)
and its phase children (#69–#81). Model choice is in
[hybrid architectures](09-hybrid-architectures.md).

Like everything above the [seam](01-architecture-seam.md), the pipeline is **portable** —
no `mlx`/`torch` import. It produces shards; the trainer consumes them.

## Why this shape

- **One framework (`datatrove`).** Every source shares streaming readers, filters, MinHash
  dedup, tokenization, and sharded writers, so the same flow scales from a laptop to a Slurm
  cluster. Alternatives considered and rejected for this POC: AI2 Dolma toolkit, NVIDIA NeMo
  Curator (#69).
- **One schema.** Every source maps into `{text, source, lang, license, meta}` so downstream
  stages treat all sources identically.
- **Two artifacts, kept separate.** Cleaned/deduped **text shards** (durable, re-mixable)
  live under a different R2 prefix from the **tokenized training shards**. Dedup/decontam are
  the expensive work — run once, then re-blend / re-tokenize cheaply.
- **Clean-license throughout.** The commercial-use problem is *distilling responses from
  commercial models*; avoid that one move and permissive/verifiable data is fine. Acceptable:
  MIT / Apache-2.0 / BSD / ODC-BY; CC-BY-SA flagged in the `license` field (attribution +
  share-alike).

## Infrastructure (Phase 1, #80)

- **R2** for durable storage (no egress fees). Target ~1–2 TB working size, growing toward
  ~4 TB for the full pool. Cleaned-text and tokenized shards under separate prefixes.
- **RunPod CPU pod** for processing (ingest/clean/dedup/tokenize); GPU pods are training-only.
  `datatrove` + `s3fs` + tokenizer deps installed there; `datatrove`'s S3 reader/writer point
  at the R2 endpoint through `s3fs`.
- **Region:** RunPod network volumes are region-locked — keep the RunPod region
  network-close to R2 so the train-time pull is free and fast.
- **Secrets:** R2 key/secret and the HF token live in RunPod secrets / env, never committed.
- **Cost shape:** R2 Class A operations cost per million, so prefer **few large shards** over
  many small files (see tokenize/shard below).

## The six stages (Phase 2–3, #69–#74)

| Stage | What | Issue |
|---|---|---|
| 1 Ingest | Stream FineWeb-Edu + parquet supplements from HF; resolve Stack `blob_id`s from Software Heritage S3, split into code vs docs-languages streams | #70, #71 |
| 2 Normalize | Map every source into `{text, source, lang, license, meta}` | #69 |
| 3 Filter | Light heuristics on text; permissive-license-only on code; drop autogenerated/minified; scrub secrets/PII | #72 |
| 4 Dedup | Exact dedup, then MinHash LSH — **cross-source** (the heavy stage; sizes the CPU pod) | #73 |
| 5 Decontaminate | Strip n-gram overlap (13-gram + 7-gram) against the eval benchmarks (#79) | #73 |
| 6 Blend | Core sets at natural size; supplements 2–3 passes; oversample TS/Rust/SQL; keep any source < ~4 passes | #74 |

### Sources

| Source | Role | License | Issue |
|---|---|---|---|
| FineWeb-Edu (~1.3T tok) | curated web-text spine | ODC-BY | #70 |
| The Stack (permissive-filtered via SWH) | code + documentation languages | repo licenses (SWH-resolved) | #70 |
| FineWiki | clean encyclopedic knowledge | CC-BY-SA (flag) | #71 |
| OpenWebMath (~14.7B) | math web text + reasoning | open (verify) | #71 |
| arXiv subset | scientific/technical writing | mixed, mostly CC | #71 |
| Stack Exchange / SO | technical problem-solving | CC-BY-SA (flag) | #71 |
| Cosmopedia (~25B synthetic) | textbook-style density | Apache-2.0 (Mixtral) | #71 |

The natural blend lands near **73% text / 27% code**; upsample code (and TS/Rust/SQL) if more
code weight is wanted. The Stack needs the **Software Heritage / INRIA agreement** — block on
it early (only the IDs stream from the Hub; the blobs come from SWH S3).

## Tokenize + shard (Phase 3, #74)

> **Superseded by the distillation pivot ([10](10-distillation.md)).** The tokenizer is now
> **fixed to Qwen2.5 (vocab 151,646)** by the conversion teacher — it intentionally *exceeds*
> the uint16 bound, so packing is **uint32** (#90, now implemented: dtype-aware `pack.py` +
> relaxed `MambaConfig.validate()`). The StarCoder2/uint16 reasoning below is the historical
> record of the from-scratch plan; the #75 production-reserve corpus reuses these
> clean-corpus *stages* but under the **same Qwen2.5 tokenizer + uint32 packing** — the
> tokenizer lock is POC *and* production, so StarCoder2/uint16 is fully superseded.

- **Tokenizer:** reuse a mixed prose+code tokenizer, chosen with the **uint16 packing bound
  (`vocab < 65536`)** in mind — enforced by `src/data/pack.py` and `MambaConfig.validate()`
  (see [data pipeline](04-data-pipeline.md)). **StarCoder2** (vocab ~49,152) **fits**;
  **Llama-3** (vocab ~128,256) **does not** — choosing it forces uint32 packing (≈2× shard
  storage) and relaxing the bound. This is the linked storage/tokenizer decision, the same
  shape as the OLMo-vs-OLMo-2 choice at POC scale.
- **Pack:** fixed-length sequences of **8192** with document-boundary markers; **reset the
  SSM state at boundaries** so recurrent state can't bleed across docs (the packing hazard —
  see #68 and [hybrid architectures](09-hybrid-architectures.md)).
- **Shard:** high-hundreds-of-MB to low-GB shards (few large files → fewer R2 Class A ops),
  written to R2 alongside the cleaned text.

## R2 storage layout

```
r2://<bucket>/
  cleaned/   <source>/*.parquet   # Stage-4/5 output: durable, re-mixable text shards
  tokens/    <tokenizer>/<seqlen>/*.bin   # Stage-6 output: training shards (uint16, or uint32 for Qwen2.5)
  ckpt/      <run>/...            # checkpoints synced from RunPod (RunPod is not durable)
```

Dedup/decontam produce `cleaned/` once; `tokens/` is cheap to regenerate when the blend,
tokenizer, or sequence length changes.

## Training flow (Phases 4–5, #81 / #75)

- **Phase 4 (#81):** the 100M smoke run — sync the ~20 GB subset from R2 to a 50 GB network
  volume on a cheap GPU pod (T4/L4), train via `scripts/train.py --backend cuda`, confirm a
  decreasing val-perplexity curve, **checkpoint back to R2**, and run the retrieval probes
  (#79). Validates the whole plumbing before paying for big cards.
- **Phase 5 (#75):** the 1B → 2B → 4B tiers per the #65 sizing table. **8-bit Adam** when
  VRAM-tight; **spot instances + frequent checkpointing** for 2B/4B; back up intermediate
  checkpoints to R2. Reuses the portable loop + two-concern [checkpointing](05-training.md);
  success is a smoothly decreasing held-out val-perplexity curve at scale.

## Post-training (Phase 6, #76–#78)

Clean-license, extending the M9 SFT+DPO machinery (#57). See those issues for dataset detail;
the licensing reframe is the load-bearing part:

- **SFT (#76):** OASST1 (multi-turn, Apache-2.0), Dolly (CC-BY-SA, flagged), FLAN, plus a
  small hand-authored in-format set; synthetic top-up only from *permissive* open models.
- **DPO (#77):** OASST1 rankings, SHP, and **on-policy self-generated** pairs (inherently
  clean). HH-RLHF excluded despite its MIT license — its responses are commercial-model
  output.
- **RLVR / GRPO (#78):** the cleanest stage — reward comes from a *verifier* (compiler, test
  runner, exact match), not licensed data, so only problems + tests are needed. Math first
  (exact-match, no sandbox), then code; ≥5 tests per coding problem; author own TS/SQL tasks
  (a SQL task = schema + question + expected result set).
